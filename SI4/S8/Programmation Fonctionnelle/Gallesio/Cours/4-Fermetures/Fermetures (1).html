Notion d’environnement (1) — Rappel

Soit la définition suivante:

(define f
  (let ((x 100))
    (display x)
    (λ (n)
      (+ x n))))

Regardons les associations lors de l’appel «(f 1)»

   f ⟶  (λ (n) (+ x n))          global
   ------------------------------------------
   x ⟶  100                      let
   ------------------------------------------
   n ⟶  1                        fonction

Une fonction Scheme, c’est donc:

  • le code de la fonction

  • ET un environnement associé



Notion d’environnement (2) — Rappel

Soit

(define (monnaie->euro taux)
  (λ (n) (/ n taux)))

On peut alors définir:

> (define dollar->euro  (monnaie->euro 1.23))
> (define livre->euro   (monnaie->euro 0.89))
> ....

  • Les fonctions ont le même code associé:

    (λ (n) (/ n taux))

  • Par contre, elles ont capturé un environnement différent:

    • taux ⟶ 1.23 (pour dollar->euro)
    • taux ⟶ 0.89 (pour livre->euro)
    • …


Notion d’environnement (3)

Les environnements peuvent s’«emboîter»

(let ((x 10))
  (let ((y 20))
    (let ((z 30) (w 40))
      (+ x y z w n))))

Le corps de ce let

  • a pour code (+ x y z n)

  • a pour environnement

    ( ((z 30) (w 40))                     ; let le plus interne
  ((y 20))                            ; let intermédiaire
  ((x 10))                            ; let externe
  …                                   ; tous les symboles globaux
)


Affectation

  • On ne peut affecter qu’une variable déclarée au préalable (par un define ou dans un let).

  • L’affectation est réalisée par la forme spéciale «set!»

    > (define x 10)
;; x est définie. On peut maintenant changer sa valeur
> (set! x 20)
> x
20

    ou encore

    > (let ((var 1))
    (set! var (* 2 var))
    var)
2

On essaiera de limiter l’utilisation de l’affectation aux situations où est elle strictement nécessaire.



Environnement et affectation (1)

Soit la fonction suivante

(define f
  (let ((c 1))
    (λ(n)
      (+ n c))))

Ici,

  • «f» est une fonction (puisque le résultat du let est (λ(n) ...))

  • «c» joue un rôle similaire à celui d’une variable statique de C ou de Java:

    • a la même durée de vie que la fonction «f»
    • n’est visible (et donc utilisable) que dans la fonction «f»

⟹

  • Si on modifie «c» lors d’un appel, la valeur modifiée sera accessible au prochain appel de «f».


Environnement et affectation (2)

Soit la fonction

1: (define f
2:   (let ((c 1))
3:     (λ(n)
4:       (set! c (+ c 1))
5:       (+ n c))))

A chaque appel, la variable «c» est incrémentée

> (f 1)
3
> (f 1)
4

Attention:

(define g
  (λ(n)                ;; on a inversé les lignes 2 et 3 précédentes
    (let ((c 1))       ;; le let est maintenant à l'INTERIEUR de la λ
      (set! c (+ c 1))
      (+ n c))))

> (g 1)                ;; un nouveau «c» est crée à chaque appel
3
> (g 1)
3


Define et affectation (1)

On peut toujours écrire une définition «en 2 fois»:

(define x 10)

peut se récrire en:

(define x #f)         ;; ou n'importe quelle autre valeur
...
(set! x 10)

Donc, la fonction précédente peut se récrire en

(define f #f)

...

(set! f (let ((c 1))
          (λ(n)
            (set! c (+ c 1))
            (+ n c))))


Define et affectation (2)

Dans la forme,

(define f #f)

...

(set! f (let ((c 1))
          (λ(n)
            (set! c (+ c 1))
            (+ n c))))

le symbole «f» est aussi accessible depuis le corps du let.

⟹ On peut donc aussi récrire cette fonction sous la forme suivante:

(define f #f)

...

(let ((c 1))
  (set! f (λ(n)
            (set! c (+ c 1))
            (+ n c))))


Define et affectation (3)

On a vu que:

  • Une fonction capture les variables d’un let englobant

  • On peut affecter une fonction à une variable globale

  • Les variables du let ne sont accessibles que depuis la fonction qui les capture (notion de variable privée)

⟹

On peut avoir deux (ou plusieurs) fonctions qui partagent une (ou plusieurs) variables privées.



Define et affectation (4)

Exemple:

(define incr #f)
(define decr #f)

(let ((c 0))
  (set! incr (λ()
               (set! c (+ c 1))
               c))
  (set! decr (λ()
               (set! c (- c 1))
               c)))

;; Utilisons incr et decr
> (incr)   ;; ⟹ 1
> (incr)   ;; ⟹ 2
> (decr)   ;; ⟹ 1

Ce qu’on a:

  • Deux fonctions qui peuvent accéder à une variable commune
  • La variable commune n’est accessible qu’à travers ces deux fonctions

Cela correspond à la notion de Type Abstrait de Données (ou ADT)



ADT Stack (1)

On veut construire le Type Abstrait de Données Stack

Les primitives que l’on doit construire:

  • push

  • pop

  • print-stack

  • empty-stack?



ADT Stack (2)

Construisons maintenant l’ADT Stack

(define push #f)
(define pop  #f)
(define print-stack #f)
(define empty-stack? #f)

(let ((S '()))
  (set! push (λ(v) (set! S (cons v S))))

  (set! pop (λ()
              (if (empty-stack?)
                  (error "Empty stack")
                  (let ((top (car S)))
                    (set! S (cdr S))
                    top))))

  (set! print-stack (λ() (display S)))

  (set! empty-stack? (λ() (null? S))))


ADT Stack (3)

  • On a bien un ADT

  • Mais, deux problèmes majeurs avec cette version

    • Pollution de l’espace de noms.
      Si on voulait généraliser:

      • empty-stack?

      • empty-queue?

      • empty-deque?

      • …

    • Une seule pile pour tout le programme

Nous allons essayer de résoudre ces deux problèmes.



Pollution de l’espace de noms (1)

Regardons le code suivant:

(let ((S '()))
  (define push   (λ(v) ...))
  (define pop    (λ() ...))
  (define print  (λ() ...))
  (define empty? (λ() ...))

  ;;; <<<ICI>>>

)

Ce qu’on peut dire:

  • Les fonctions «push», «pop», … ne sont accessibles que dans le let;
  • Les fonctions «push», «pop», … «voient» la variable S;
  • Du point <<<ICI>>>, on voit les fonctions «push», «pop», …


Pollution de l’espace de noms (2)

En plaçant une fonction au point <<<ICI>>>, on peut:

  • appeler chacune des fonctions internes
  • et, par conséquent, accéder (indirectement) à la variable interne.

On obtient:

(define switch
  (λ (op . arg)
    (cond
      ((eq? op 'push)   (push (car arg)))
      ((eq? op 'pop)    (pop))
      ((eq? op 'print)  (print))
      ((eq? op 'empty?) (empty?))
      (else             (error "Unknown operation: " op)))))


Pollution de l’espace de noms (3)

Soit le code:

(let ((S '()))
  (define push   (λ(v) ...))
  (define pop    (λ() ...))
  (define print  (λ() ...))
  (define empty? (λ() ...))

  (define switch
    (λ(op . arg)
      ...))
  switch)
  • La valeur de ce «let» est la fonction «switch» qui sert d’aiguillage.
  • Cette fonction est le moyen d’accéder aux fonctions et données internes.
Note:
Les autres fonction ne peuvent pas être appelées directement

⟹ on a résolu le problème de pollution de l’espace de noms.



Première version de l’ADT Stack (1)

(define stack
  (let ((S '()))
    (define push ...)
    (define pop ...)
    (define print ...)
    (define empty? ...)

    (define (swicth op . arg)
      ...))

    ;; On renvoie la fonction switch
    switch)

La valeur de stack est la fonction d’aiguillage: swicth.

Si on veut, on peut se passer de la nommer:

(define stack
  (let ((S '()))
    (define push ...)
    (define pop ...)
    (define print ...)
    (define empty? ...)

    (λ (op . arg)   ;; Résultat du let: la fonction d'aiguillage
       ...)))


Première version de l’ADT Stack (2)

On avait:

(define stack
  (let ((S '()))
    (define push ...)
    (define pop ...)
    (define print ...)
    (define empty? ...)

    (λ (op . arg)   ;; Résultat du let: la fonction d'aiguillage
       ...)))

Quelques utilisations:

> (stack 'empty?)
#t
> (stack 'push 1)
> (stack 'push 2)
> (stack 'empty?)
#f
> (stack 'pop)
2
> (stack 'print)
(1)


Première version de l’ADT Stack (3)

Code complet:

(define stack
  (let ((S '()))
    (define push (λ(v)
                   (set! S (cons v S))))
    (define pop   (λ()
                    (if (empty-stack?)
                        (error "Empty stack")
                        (let ((top (car S)))
                          (set! S (cdr S))
                          top))))
    (define print (λ()
                    (display S)))
    (define empty? (λ()
                     (null? S)))

    (λ (op . arg)
      (cond
        ((eq? op 'push)   (push (car arg)))
        ((eq? op 'pop)    (pop))
        ((eq? op 'print)  (print))
        ((eq? op 'empty?) (empty?))
        (else             (error "Unknown operation: " op))))))


Deuxième version de l’ADT Stack (1)

  • Problème de la pollution de l’espace de noms: résolu

  • mais on a toujours qu’une seule pile dans le programme.

    • Il faut définir le «générateur» de pile make-stack:
      • à chaque exécution ⇒ construction d’une nouvelle pile;
    • Implémentation simple:
      • il suffit d’embarquer le «let» dans la fonction make-stack:
    (define (make-stack)  ;;; SEULΕ MODIFICATION: stack → (make-stack)
  (let ((S '()))
     ...))

On a donc,

> (define p1 (make-stack))
> (define p2 (make-stack))
> (p1 'push 1)
> (p1 'empty?)
#f
> (p2 'emty?)
#t


Deuxième version de l’ADT Stack (2)

Code complet:

(define (make-stack)   ;;; SEULΕ MODIFICATION: stack → (make-stack)
  (let ((S '()))
    (define push (λ(v)
                   (set! S (cons v S))))
    (define pop   (λ()
                    (if (empty-stack?)
                        (error "Empty stack")
                        (let ((top (car S)))
                          (set! S (cdr S))
                          top))))
    (define print (λ()
                    (display S)))
    (define empty? (λ()
                     (null? S)))

    (λ (op . arg)
      (cond
        ((eq? op 'push)   (push (car arg)))
        ((eq? op 'pop)    (pop))
        ((eq? op 'print)  (print))
        ((eq? op 'empty?) (empty?))
        (else             (error "Unknown operation: " op))))))


Version finale de l’ADT Stack (1)

Scheme propose la forme case qui permet de simplifier l’écriture du cond précédent.

(λ (op . arg)
  (cond
    ((eq? op 'push)   (push (car arg)))
    ((eq? op 'pop)    (pop))
    ((eq? op 'print)  (print))
    ((eq? op 'empty?) (empty?))
    (else             (error "Unknown operation: " op))))

peut se réécrire en

(λ (op . arg)
  (case op
    ((push)   (push (car arg)))
    ((pop)    (pop))
    ((print)  (print))
    ((empty?) (empty?))
    (else     (error "Unknown operation: " op))))

On pourrait même avoir:

(λ (op . arg)
  (case op
    ((push empiler)    (push (car arg))) ;; si op ∈ {push, empiler}
    ((pop dépiler)     (pop))            ;; si op ∈ {pop, dépiler}
    ...))


Version finale de l’ADT Stack (2)

Avec la forme case:

(define (make-stack)                ;;; version finale.
  (let ((S '()))
    (define push (λ(v)
                   (set! S (cons v S))))
    (define pop   (λ()
                    (if (empty-stack?)
                        (error "Empty stack")
                        (let ((top (car S)))
                          (set! S (cdr S))
                          top))))
    (define print (λ()
                    (display S)))
    (define empty? (λ()
                     (null? S)))

    (λ (op . arg)
      (case op
        ((push)   (push (car arg)))
        ((pop)    (pop))
        ((print)  (print))
        ((empty?) (empty?))
        (else     (error "Unknown operation: " op))))))

Pour la suite, on utilisera en général un case pour la fonction d’aiguillage.



Des ADTs aux Objets

Avec les ADTs, on a

  • des données privées
  • des fonctions privées qui peuvent accéder au données
  • possibilité d’avoir plusieurs instances de l’ADT

Dans les langages objets à base de classes, une classe a

  • des données privées
  • des fonctions privées qui peuvent accéder au données (méthodes)
  • possibilité d’avoir plusieurs instances de la classe (objets)
  • possibilité de construire une classe à partir d’une autre (héritage)

Pour passer des ADTs aux objets, il ne nous reste donc qu’à savoir implémenter l’héritage.

Nous verrons que

Closures are functions with data. Objects are data with functions.
